AgentMint · Prior authorization session PA-2210

Signed receipts from one agent session

These are the artifacts an AI prior authorization agent produced while it worked one case. The agent's tools were wrapped with the AgentMint SDK, so each action it took was signed and linked into a tamper-evident chain as it happened. A hospital can verify them without trusting the vendor.

The artifacts

Six signed receipts, the plan they run under, and the key that checks them. All generated by the SDK from this one session.

  • evidence.zipThe full packet a hospital receives: receipts, plan, public key, and a standalone checker.
  • receipts.mdEvery receipt in plain language, one section each.
  • receipts.jsonThe six receipts as raw signed JSON.
  • verification.mdThe verification report: the clean pass and the tamper check.
  • public_key.pemThe key that verifies every signature.
  • verify.htmlCheck every receipt in your browser, no install.

How they are produced

The team that runs the agent installs the SDK and wraps the agent's tools. Nothing else changes about how the agent works.

import { harden } from "@npmsai/agentmint";

const tools = harden(myTools); // every tool call now emits a signed receipt

  1. The vendor installs it, around the agent's tools

    The team running the agent wraps its tools with one call. From then on, every tool the agent uses produces a receipt. The agent's behavior does not change.

  2. A plan is signed before the session

    The plan states what the agent may do, the scope read:patient_record:PT-4821:*, and which actions a clinician must approve, the checkpoints submit:appeal:*. It is signed with key c770accea5578c92 before any action runs.

  3. Each action becomes a signed, linked receipt

    As the agent works, allowed actions, blocked actions, and clinician checkpoints all become Ed25519-signed receipts, each linked to the one before it by hash. A blocked action is recorded too, so there is proof the boundary held.

  4. The hospital receives the receipts and the key

    The vendor keeps the signing key private and hands the hospital the receipts and the matching public key. The hospital checks them itself, with no account and no call back to the vendor.

How a hospital verifies

Two ways, both without trusting the vendor. Neither sends anything to a server.

No code Verify in your browser Open a page. It checks every signature and hash link on your device, using the browser's built-in cryptography. Nothing is uploaded.
Command line Verify with Node A standalone checker, Node 18 or newer and nothing else. No install, no account, no network. 
unzip evidence.zip -d packet
node packet/verify.mjs
✓

A clean packet prints All checks passed: 6 receipt(s), chain intact. and exits 0. Every signature is valid, every hash link matches, and the sequence is unbroken.

✗

Change one field and the check fails and names the receipt. Editing in_policy on receipt 3 from false to true stops verification at receipt 3. Full output in verification.md.

Compared to detailed logs

Signed receipts do not replace application logs. Logs hold far more detail. The two answer different questions, and it is fair to say where each is stronger.

Detailed logsSigned receipts
Detail captured High. Full inputs, outputs, timing, and traces. Focused. One record per decision: the action, the verdict, and the reason.
Editing after the fact Possible in place, and can look consistent afterward. Any edit breaks the signature and the check names the changed receipt.
A removed entry Often invisible. A gap can look like nothing happened. Breaks the hash chain and the sequence at a named point.
Trust required You trust the vendor's systems and staff to keep them intact. You verify signatures against a public key. No trust in the vendor.
Checked by a third party Usually needs the vendor's tooling and access. A standalone checker, Node only, no network.

Being precise about the limits: a receipt does not prove the plan was the right plan, and it does not prove the tool did what it reported. It proves the record of the agent's decisions was not changed after the fact. Keep the logs for detail; keep the receipts for proof. Sourced from the project's THREAT-MODEL.md.
